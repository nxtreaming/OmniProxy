# OmniProxy - 统一 AI 令牌调度与无感代理网关

## 1. 项目背景与目标

**OmniProxy** 是一款专为本地开发者设计的轻量级 AI 代理工具。它通过在本地启动一个“透明反向代理”服务器，劫持原本发往特定 AI 厂商（如 Codex, OpenAI, Claude 等）的请求，并根据预设的 Token 池和额度规则进行智能调度。

**核心目标：**

- **无感切换**：在 `codex`、`claude code`等工具运行期间，自动处理额度耗尽或 429 报错，无需重启客户端即可完成账号切换。
- **额度预警**：实时监测每个账号的剩余额度，当额度低于阈值（如 15%）时自动切换至下一个可用账号。
- **多厂商支持**：架构上兼容多种 AI API，支持不同厂商 Token 的统一管理。
- **极致轻量**：采用 Wails (Go + Vue) 构建，运行开销低，启动速度快。

## 2. 核心功能要求

### 2.1 代理层 (Backend - Go)

- **透明转发**：拦截本地端口请求，将请求头中的 `Authorization` Token 替换为池中的有效 Token，并转发至真实 API。
- **响应监测**：解析 API 返回的响应头（如 `x-ratelimit-remaining`），更新本地 Token 额度。
- **自动重试**：捕获 429 或 5xx 错误，在代理层内部自动重传请求，确保客户端侧感知不到中断。
- **Token 池管理**：支持 Token 的增删改查，记录每个 Token 的状态（活跃、耗尽、失效）和最后消耗时间。

### 2.2 控制面板 (Frontend - Vue 3)

- **可视化仪表盘**：展示当前正在使用的 Token 及其额度曲线。
- **账号管理列表**：
  - **排序规则**：新添加的账号默认出现在列表最顶部。
  - **表单校验**：账号名称必填且不可重复，Token 格式校验。
- **实时日志**：展示代理转发的请求状态、耗时以及切换记录。
- **全局设置**：配置代理端口、额度切换阈值（默认 15）等。

### 2.3 UI/UX 设计规范

- **视觉风格**：采用现代化扁平化设计，大量使用圆角矩形（Rounded Rectangles）。
- **交互体验**：侧边栏宽度可调节，支持深色/浅色模式切换。
- **状态反馈**：Token 状态通过不同颜色的 Tag 指示（绿色-正常，黄色-低额度，灰色-耗尽，红色-无效）。

## 3. 技术栈选型

- **框架**：[Wails v2](https://wails.io/) (Go + Vue 3 + Vite)
- **后端语言**：Go (1.20+) - 负责网络 I/O、反向代理、本地存储 (SQLite/JSON)
- **前端语言**：TypeScript, Vue 3 (Composition API)
- **UI 组件库**：Element Plus 或 Naive UI
- **状态管理**：Pinia
- **网络库**：Go `net/http/httputil` (反向代理核心)

## 4. 实现步骤

### 第一阶段：项目初始化与环境搭建

1. 安装 Wails CLI：`go install github.com/wailsapp/wails/v2/cmd/wails@latest`
2. 初始化项目：`wails init -n omniproxy -t vue-ts`
3. 前端引入组件库（如 Element Plus）并配置基础布局（侧边栏、主视图区域）。

### 第二阶段：后端代理引擎实现 (Go)

1. **定义数据结构**：设计 `Token` 结构体，包含 `ID`, `Name`, `Provider`, `TokenValue`, `Remaining`, `Status` 等字段。
2. **实现 Token 管理器**：编写 Token 的增删改查逻辑，并持久化到本地（推荐使用 `buntdb` 或简单的 JSON 文件）。
3. **构建代理核心**：
   - 使用 `httputil.NewSingleHostReverseProxy` 创建代理实例。
   - 重写 `Director` 方法：动态注入 Token。
   - 重写 `ModifyResponse` 方法：解析 Header 获取额度并同步到前端。
   - 自定义 `Transport` 处理重试逻辑：捕获 429 错误并切换 Token 重发。

### 第三阶段：前端管理界面开发 (Vue)

1. **账号列表页面**：实现列表展示，确保新数据置顶。加入“手动测速/验证”按钮。
2. **表单组件**：实现账号添加/编辑弹窗，加入非空校验逻辑。
3. **Wails 绑定**：在前端通过 Wails 自动生成的 JS/TS 绑定，直接调用后端的增删改查方法。
4. **仪表盘展示**：利用图表库（如 ECharts）展示 Token 消耗趋势。

### 第四阶段：联调与优化

1. **代理配置**：在 `codexcli` 中设置 API BaseURL 或 Proxy 为 `http://127.0.0.1:3000`。
2. **无感切换测试**：通过压力测试或模拟额度耗尽，观察代理层是否能平滑切换，并且客户端不抛出报错。
3. **性能调优**：优化 Wails 窗口启动速度，确保代理层在高并发下的稳定性。

## 5. 核心逻辑伪代码 (Go)

```
// 核心代理重写逻辑
proxy.ModifyResponse = func(resp *http.Response) error {
    // 1. 获取剩余额度
    remaining := resp.Header.Get("x-ratelimit-remaining-tokens")
    
    // 2. 如果额度 < 阈值，标记当前 Token 耗尽
    if val, err := strconv.Atoi(remaining); err == nil && val < 15 {
        tokenPool.MarkExhausted(currentTokenID)
    }
    return nil
}

// 重试逻辑 (Transport)
type RetryTransport struct {
    base http.RoundTripper
}

func (t *RetryTransport) RoundTrip(req *http.Request) (*http.Response, error) {
    // 备份请求体以便重试
    var bodyBytes []byte
    if req.Body != nil {
        bodyBytes, _ = io.ReadAll(req.Body)
        req.Body = io.NopCloser(bytes.NewBuffer(bodyBytes))
    }

    resp, err := t.base.RoundTrip(req)
    
    // 拦截 429 报错，进行无感切换
    if resp != nil && resp.StatusCode == http.StatusTooManyRequests {
        // 更换 Token 并重试
        newToken := tokenPool.GetNextActive()
        req.Header.Set("Authorization", "Bearer "+newToken)
        
        // 恢复请求体
        if bodyBytes != nil {
            req.Body = io.NopCloser(bytes.NewBuffer(bodyBytes))
        }
        
        return t.base.RoundTrip(req)
    }
    return resp, err
}
```
